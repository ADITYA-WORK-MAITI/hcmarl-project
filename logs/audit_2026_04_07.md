### hcmarl/three_cc_r.py

**[C-1] `step_euler` silently corrupts state via clip-and-renormalize (lines 278–283).**
After the Euler update, the code does `np.clip(x_new, 0, 1)` then `x_new /= total`. The ODE system already conserves mass exactly (Theorem 3.2). Clipping + renormalizing is not a projection onto the simplex — it is a multiplicative rescaling that arbitrarily redistributes mass when any component goes negative. Concretely: if `kp=10` (default), `TL=0.5`, `MA=0`, then `C_nom=5`, and one Euler step of `dt=1 min` produces `dMA = C - F·MA ≈ 5`, so `MA_new ≈ 5`. Clipped to 1, then total ≈ 1 + something, then divided. The "fatigue" and "resting" compartments are now whatever the renormalization happens to make them — not what the ODE says. **Every Phase B training run with default `kp=10`, `dt=1` will be numerically meaningless** because the env is supposed to call this `step_euler`. Fix: either (a) reduce `dt` (sub-stepping) so Euler stays in `[0,1]` without clipping, (b) use the existing `solve_ivp` path inside `simulate()` for the per-step update, or (c) project onto the simplex `{x ≥ 0, sum=1}` properly (closed-form Euclidean projection). Option (b) is safest and the math doc’s pipeline (§7.3 step 6) says "Integrate ODEs", not "Euler-step them".

**[C-2] Default `kp=10.0` is incompatible with `dt=1 min` Euler stepping.**
For shoulder `F=0.0146/min`, the natural relaxation time is ~70 min; `kp=10/min` is two orders of magnitude faster than the plant. Combined with [C-1], the very first env step from a fresh worker on any nontrivial task overshoots `MA=1` and the state becomes garbage. Even with substepping, `kp=10` is unjustified by the math doc — Eq 35 just says `kp > 0`. Recommend `kp ≈ F` (≈0.01–0.02) or whatever value is calibrated to reproduce Frey-Law endurance times. **Until calibrated, do not run training.**

**[S-1] `simulate()`: `C_values` collected inside `rhs` is dead (line 322, 330).**
`C_values.append(C)` inside the RHS callback is appended at every internal RK45 stage (non-monotonic, non-uniform times), then never returned — the function recomputes C at `sol.t` afterwards. Harmless but reveals the RHS is being called many times per output point; if you ever use `C_values` you'll get garbage. Delete or replace with the post-hoc loop already present. Minor.

**[N] Conservation check `__post_init__` uses `1e-6` tol; ODE conservation is exact algebraically, so this is fine for hand-built states but will trip if you ever build state from raw simulation output without renormalizing.**

### hcmarl/ecbf_filter.py

**[N] ECBF constraint construction (lines 287–293) matches Eq 18 exactly. Verified by hand expansion.**

**[N] `cbf_upper_bound` (line 213) matches Eq 23. `min_rest_duration_bound` matches Eq 30. `psi1_jump_at_rest` matches Eq 28.**

**[S-2] `filter_analytical` advertised as "equivalent to the QP solution when C_nominal is scalar" (line 370) — only true when at least one upper bound is ≥ 0.**
If both `ub_ecbf < 0` and `ub_cbf < 0` (deeply infeasible state), `min(C_nom, ub_ecbf, ub_cbf) < 0`, then `max(0, …) = 0`, which matches the QP infeasibility fallback (Remark 5.13). So the docstring is actually correct in outcome, but the path "infeasible → 0" is conflated with "feasible → C_safe", and you lose the ability to log infeasibility. If you ever use `filter_analytical` in the env hot loop (which you should, for speed), add an explicit infeasibility flag — otherwise you cannot tell mandatory rest from voluntary rest in your training logs. Serious for paper-quality diagnostics, not for safety.

**[S-3] `filter()` returns `C_filtered = 0.0` on infeasibility (line 335) but `was_clipped` is computed only by comparing to `C_nominal`. If `C_nominal == 0`, `was_clipped=False` even though the QP was infeasible.** This will under-report mandatory-rest events in your evaluation tables. Add an `infeasible: bool` field to `ECBFDiagnostics`.

**[S-4] No assertion that the filtered `C` actually satisfies `h ≥ 0` and `h2 ≥ 0` after the next state step.**
The ECBF guarantees forward invariance *in continuous time*. With Euler stepping (see C-1), it does not guarantee invariance in discrete time. Without an explicit post-step check, you may report "ECBF safe" while the discretized trajectory crosses the barrier. Add a debug-mode assertion `assert ecbf.h(new_state.MF) >= -tol` after `step_euler` and surface violations. Without this, your safety claims are unverified empirically.

### hcmarl/nswf_allocator.py

**[C-3] `_solve_greedy` (lines 307–360) is invoked when `N > 8 or M > 8` and is a heuristic, not optimal.**
The paper's central claim in §6 is that you maximize the NSWF (Eq 33). Your `scaling_n12.yaml` config has N=12. For that config the allocator silently switches to a greedy heuristic that returns suboptimal allocations. **You will report "NSWF allocation" in the paper while actually running greedy assignment.** A reviewer who reads `nswf_allocator.py:226` will catch this in 30 seconds. Fix: log-NSWF with positive surpluses is a linear assignment problem (sum of log-surpluses) — use `scipy.optimize.linear_sum_assignment` on the cost matrix `-log(surplus)` (with rest as a self-loop column per worker). This is exact, polynomial, and standard. The current `N≤8` exact backtracker can also be replaced by Hungarian to unify the code path. **Must fix before any scaling experiment.**

**[S-5] `disagreement_utility` raises `ValueError` for `MF < 0` but returns `+inf` for `MF ≥ 1` (lines 115–118).**
Inconsistent. If you trust the env to keep MF in [0,1], make both raise; if you accept boundary cases, make both return inf/0. The asymmetry will surface as a confusing crash mid-training when [C-1]'s renormalization briefly produces a slightly negative MF.

**[S-6] Greedy does not retry workers whose first attempt was blocked by an earlier higher-gain pick.**
A worker can end up at rest while a still-available task with positive surplus was never tried for them. Strictly worse than Hungarian. Rendered moot once [C-3] is fixed.

**[N] `surplus_matrix = utility_matrix - D[:, None]` (line 218) correctly subtracts disagreement utility per worker.**

### hcmarl/pipeline.py

**[S-7] `from_config` default `theta_max = min(theta_min_max * 1.1, 0.95)` (line 368).**
For shoulder this gives `0.627 * 1.1 = 0.690`. Operationally this means you allow shoulder fatigue up to 69% MVC before the filter activates — this is *above* the design threshold (62.7%) by design (Eq 26 is satisfied), but it leaves only ~6 percentage points of margin. Your choice of `max` is defensible (most-fatigued muscle dominates) but is **an undocumented modeling decision** that affects every allocation. ### hcmarl/mmicrl.py

**[C-4] `ConstraintNetwork.train_on_demos` is circular (lines 456–542).**
The labels `y` are derived from the 90th percentile of MF in the demos themselves (line 486–491). The network is then trained to predict those labels, and `theta_max` is extracted by sweeping MF until the network's output crosses 0.5 (line 533–535). By construction this recovers ~the input percentile. There is **no inverse-RL signal** — no constraint is being inferred from the agent's choices, only from the empirical MF marginal. This is not Malik et al. (2021). A reviewer who understands ICRL will see the percentile→network→percentile loop and reject the contribution as ICRL. Two acceptable fixes: (a) honestly relabel this as "percentile-based threshold per type" and remove all ICRL claims, or (b) implement the proper Malik objective: learn `c_θ(s,a)` to discriminate expert state-actions from a uniform/random policy's state-actions, then derive Θmax from where `c_θ` flips. **Must address before any "learned constraint" claim in the paper.**

**[S-10] `_compute_mutual_information` returns `H(z)` (hard-assignment entropy) on mode-collapse fallback (line 944–954) and labels it "mutual information".**
The docstring even calls H(z) an "upper bound on I(τ;z)". You then store the value as `results["mutual_information"]` and it ends up in your tables. When CFDE collapses, your reported "MI" is actually H(z), which can be ≫ I — silently inflating results. Either (a) return `0.0` on collapse with a `collapsed: True` flag, or (b) clamp to `min(H(z), I_estimate)` and flag. **Will be caught in review of any MI table.**

**[S-11] `_discover_types_cfde` E-step rejects reassignments that empty any type (line 882–891).**
This is a "guard against mode collapse" but it means you can never escape a bad K-means init. If K-means happens to put all aggressive workers in one cluster and the data actually has 2 types not 3, you will report 3 types regardless. This is sweep-the-collapse-under-the-rug behavior. Document explicitly in the paper that you constrain types to be non-empty, or run with model selection (BIC/elbow) over `n_types`.

**[S-12] `h_marginal` (lines 1064–1069) is the average per-dimension marginal entropy, not the joint trajectory entropy `H[π(τ)]` that Eq 9 calls for.**
For independent dimensions these are equal; for the strongly correlated MR/MA/MF channels they are not. When `λ1 ≠ λ2`, the residual term you're adding to the objective is the wrong quantity. Either (a) restrict to `λ1=λ2` (Remark 4.4 recommends this anyway) and refuse to compute `h_marginal` at all, or (b) use a proper joint-entropy estimator (k-NN, Kraskov). Recommend (a) — match the math doc's recommendation.

**[S-13] `validate_mmicrl` and `generate_synthetic_demos` exist in production module and `__main__` calls the synthetic path (line 1250).**
The user has explicitly banned synthetic data. There's a UserWarning but the function is still wired into `__main__`. Either move to `tests/` or guard behind an env var. Risk: someone (you, in a hurry) runs `python -m hcmarl.mmicrl` and gets results that look real. Serious for integrity.

**[S-14] `n_actions` auto-detect (line 1040–1044): `max(action) + 1`. If the demos never use the rest action (action == n_tasks-1), `n_actions` is undercounted, and `get_step_data`'s one-hot truncates with `min(action, n_actions-1)` (line 721) — collapsing distinct actions silently.** Pass `n_actions` from the env explicitly.

**[N] CFDE MADE mask construction (lines 38–55) follows the Germain et al. (2015) trick with the off-by-one for output. Verified by hand for D=4.**

**[M-2] `_BatchNormFlow` stores `self.batch_mean / self.batch_var` as instance attributes (line 143–145), not buffers. They are not saved on `state_dict`, and on a fresh load in eval mode the inverse pass would crash. Since you never call inverse during training, harmless — but call it out so future-you doesn't try to sample from the flow.**

### hcmarl/agents/mappo.py

**[C-9] Centralized critic is regressed to N different targets per identical input.**
`compute_returns` produces per-agent returns `ret_a = adv_a + values` (line 142). These are flattened so the same global state `gs[t]` appears N times in `get_flat_tensors` (lines 169–174), each paired with a *different* `ret_a[t]` (one per agent, since agents have different reward streams). The critic is then trained via `mse_loss(critic(gs), ret_t)` where `gs` is duplicated and `ret_t` is per-agent. The critic provably converges to the *mean* per-agent return, not V(s). The advantage targets it then provides for the next iteration are correspondingly biased. Standard CTDE either (a) trains the critic on a single team return per timestep (sum or mean of agent rewards), or (b) uses a critic that takes (state, agent_id) as input. Current implementation does neither. **The "centralised critic" claim is technically false; it is a mean-target critic.** Fix: compute one shared return per timestep (e.g. team-mean reward) and train the critic on that, OR add an agent-id one-hot to the critic input.

**[S-19] Bootstrap value `last_val` not masked by `dones[T-1]` (line 137).**
For terminal states the bootstrap should be 0. Minor since training-loop episodes always end at `max_steps` (truncation), but if a future env terminates early the GAE will leak past the terminal.

**[S-20] `compute_returns` uses `next_val = values[t+1]` without masking by `dones[t+1]`.**
If multiple episodes ever land in one buffer flush, value estimates leak across boundaries. Currently the train loop calls `update()` once per episode so it's safe, but the contract is fragile and should be fixed to use `dones[t+1]` masks.

### hcmarl/agents/mappo_lag.py

**[C-10] Cost advantages are normalized to mean-0/std-1 BEFORE being multiplied by `lambda` (line 221).**
After normalization, `cadv_t` no longer carries information about whether average cost is above or below `cost_limit` — it always has mean ≈ 0. The actor's "Lagrangian penalty" `λ * max(surr1, surr2).mean()` then operates on a dimensionless quantity, and the dual variable λ is updated separately by `update_lambda(mean_cost - cost_limit)`. Result: λ grows when total cost exceeds the limit, but the actor's gradient is in normalized units that have nothing to do with the limit. **The Lagrangian guarantee is broken** — the dual update and the primal penalty are decoupled. Fix: do not normalize cost advantages, OR rescale λ by `cadv.std()` so the units match. This is a known footgun (see Stooke et al. 2020 §5). Without this fix, MAPPO-Lagrangian results are not comparable to PPO-Lagrangian / CPO baselines.

**[S-21] `LagrangianRolloutBuffer` recomputes `_legacy_pending` per call but stores per-agent. Same fragile contract as MAPPO buffer — minor.**

### scripts/run_baselines.py & run_ablations.py & run_scaling.py

**[C-17] `run_ablations.py` invokes `train.py --method hcmarl --config config/ablation_*.yaml` for all five ablations, but `train.py` reads ZERO of the ablation knobs.**
Verified by grep on `train.py`: the only env keys read are `n_workers`, `max_steps`, `theta_max`. NOT `muscle_groups`, NOT `dt`, NOT `kappa`, NOT `ecbf.enabled`, NOT `ecbf.alpha*`, NOT `nswf.enabled`, NOT `nswf.allocation`, NOT `mmicrl.enabled`, NOT `mmicrl.use_fixed_theta`, NOT `disagreement.divergent`, NOT `disagreement.type`, NOT `ablation`. - `ablation_no_mmicrl` produces the SAME training because `mmicrl.enabled=false` is ignored and `--mmicrl` is a CLI flag, not a config key.

This is a reviewer-killing finding. **Fix: every ablation knob must be (a) read by `train.py`, (b) propagated into the env constructor, (c) actually change a code path, (d) verified by a unit test that turning the flag off changes a metric.**

**[C-18] `scaling_n3..n8.yaml` and `scaling_n12.yaml` are identical except for `n_workers`. None of them set `kp`, `dt`, or `ecbf.kp` differently. The "scaling" study is just running the same algorithm at different N — fine in principle, but combined with [C-3] (greedy fallback at N>8) and [C-12] (broken integrator everywhere), you get a scaling curve that confounds three things at once: real scaling, allocator switch, and integrator regime change.
