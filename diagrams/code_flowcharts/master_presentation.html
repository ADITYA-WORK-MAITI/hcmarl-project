core hcmarl/utils.py utils

Stateless numerical helpers — safe_log, clip_and_normalise, load_yaml, get_logger — shared across all modules.

This is the file I write first in any project, because everything else assumes it. clip_and_normalise (line 96) clamps an array to [0,1] and renormalises it to sum to one — I call it after every ODE tick so the conservation law MR + MA + MF = 1 never silently drifts. From the same file, safe_log (line 115) is the trick that lets the NSWF allocator take log(surplus) without falling off a cliff when a worker is fully fatigued and the surplus collapses to zero. Around those two numerical primitives sit load_yaml (line 50), which hydrates configs into dicts for the training script and every launcher, and get_logger (line 23), which returns a consistently formatted Python logger so every module's diagnostics look the same in the terminal.

Arrows in

None — this module is a leaf. It deliberately imports nothing from the rest of the codebase, which is what makes it safe to import everywhere else without creating cycles.

Arrows out

From here, safe_log walks down into nswf_allocator.py:26, which is the only place in the codebase where log is taken of a quantity that can legitimately reach zero — without this helper the allocator would crash the moment any worker reached D_i = U_ij. load_yaml and get_logger together feed pipeline.py:33, where the closed-loop simulator reads its config and emits the per-step diagnostics that the test suite later asserts against.

core hcmarl/three_cc_r.py three_cc_r

The 3CC-r ODE model: three muscle compartments (active MA, fatigued MF, resting MR) with a reperfusion multiplier, integrated forward in time.

This is the physiological heart of the framework, and it earns that title by doing exactly one thing: it tells you, for any neural drive C you push into a muscle, how the three compartments — resting MR, active MA, fatigued MF — evolve over time. The state lives inside a frozen MuscleParams dataclass that holds the three calibrated rate constants F (fatigue), R (recovery) and r (the reperfusion multiplier that boosts recovery during rest). Six muscle groups are pre-instantiated at lines 79–84 — SHOULDER(F=0.0146, R=0.00058, r=15), then ANKLE, KNEE, ELBOW, TRUNK, and GRIP with r=30 for the forearm flexors — all taken from the isometric population means in Frey-Law, Looft & Heitsman (2012) Table 2. From those three numbers the file derives the quantities the rest of the codebase actually uses: C_max = F·R/(F+R) and delta_max = R/(F+R) from Eq 6, and the safety threshold theta_min_max = F/(F+R·r) from Eq 25 that the ECBF filter inherits as its lower bound. The integrator itself is one line of physics per compartment — dMA/dt = C − F·MA, dMF/dt = F·MA − R_eff·MF, dMR/dt = R_eff·MF − C — switched by Eq 5 so that R_eff equals the slow base rate R while the muscle is loaded and jumps to R·r the instant the load drops to zero. Theorem 3.2 then guarantees that MR + MA + MF stays at one forever.

Arrows in

Nothing arrives here. This is a deliberate leaf dependency — keeping it free of internal imports means the physiology can never accidentally develop a cyclic relationship with the safety filter or the allocator that builds on top of it.

Arrows out

From here, MuscleParams and ThreeCCrState walk into ecbf_filter.py:26, because the QP that enforces safety needs the very same (F, R, r) triple to compute the barrier coefficients and the validity condition. The same dataclass plus the SHOULDER instance and the ThreeCCr integrator flow into real_data_calibration.py:37, where Path G re-fits these parameters against WSD4FEDSRM endurance times so the simulator stops running on textbook averages and starts running on measured human data. The integrator itself is consumed by pipeline.py:26–32 as the state-update step in the seven-step loop, while the get_muscle lookup is called at line 32 of pettingzoo_wrapper.py at environment construction time, once per muscle, so each worker boots with the right rate constants.

core hcmarl/ecbf_filter.py ecbf

Relative-degree-2 exponential control barrier function: solves a QP every step to project the nominal control onto the safe set.

The job of this file is to take whatever neural drive the policy proposes and project it onto the closest control that the physics still considers safe. Two barriers run in parallel — a fatigue ceiling h(x) = theta_max − MF ≥ 0 from Eq 12, and a resting floor h2(x) = MR ≥ 0 from Eq 23 — and both have to hold at every step, otherwise the projection is invalid. To enforce them, the file builds a quadratic program at line 270 that minimises ||C − C_nom||² subject to the ECBF enforcement condition from Eq 18 (−F·C + F²·MA + R_eff·F·MA − R_eff²·MF + α₁·h_dot + α₂·ψ₁ ≥ 0) and the resting-floor CBF from Eq 23 (C ≤ R_eff·MF + α₃·MR), with C ≥ 0 as a hard non-negativity constraint. The four design knobs live in an ECBFParams dataclass at line 30 — theta_max, alpha1=0.05, alpha2=0.05, alpha3=0.1 — and the file's validate() method checks the existence condition from Assumption 5.5 (theta_max ≥ F/(F+R·r)) before letting the filter run, because if that inequality fails the safe set is empty and projecting onto it is meaningless. When the condition is satisfied, Theorem 5.7 (rest-phase Nagumo invariance) takes over and guarantees the state stays inside the safe set forever during rest. The CVXPY solver at line 309 occasionally raises a "solution may be inaccurate" warning during pipeline tests, but it has been verified to be non-fatal — the returned solution still satisfies all constraints.

Arrows in

The MuscleParams and ThreeCCrState dataclasses arrive from three_cc_r.py, and they have to arrive — the QP coefficients are literally functions of F, R, r, and the current (MR, MA, MF). Without that import, this file would not know which barrier to enforce.

Arrows out

What leaves this file is a single number per step: the filtered control C_safe. It walks straight into pipeline.py, where the seven-step loop refuses to integrate the ODE until the projection has happened. C_safe replaces C_nom at step 5 of every loop iteration, which is the entire reason the framework can claim hard safety rather than statistical safety.

core hcmarl/nswf_allocator.py nswf

Nash Social Welfare allocator: maximises the log-sum of worker surpluses to find a fair task assignment under fatigue.

This file decides who does what, and it does so under a fairness criterion that has a name and a theorem behind it (Nash 1950, Kaneko & Nakamura 1979) rather than a heuristic. The starting point is the disagreement utility from Eq 32, D_i(MF) = kappa·MF²/(1−MF), which diverges as a worker's fatigue approaches one — physically saying that a near-exhausted worker has no surplus left to give to any task. From there, the surplus a worker actually gets from being assigned a task is S_ij = U(i,j) − D_i(MF_i), and the allocator only considers assignments where every surplus is strictly positive, because anything else would violate individual rationality. The Nash welfare objective from Eq 33 then asks for the assignment that maximises sum_i log(S_ij) — the log transform turns the product of surpluses into a sum, which is exactly the form RL gradients are happy with downstream. Two solvers are available: _solve_exact at line 231 enumerates every valid assignment when the problem is small, and a greedy fallback handles larger ones. The result comes back as an AllocationResult dataclass at line 302 carrying the chosen assignment, the achieved objective value, and the per-worker surplus and disagreement values that the logger and the diagnostics consume.

Arrows in

One thing arrives — safe_log from utils.py:115 — and the only reason it has to be the safe variant is that line 266 of this file is exactly where log(surplus) would otherwise crash the solver the moment a worker's surplus touched zero. The dependency is small but load-bearing.

Arrows out

The assignment dictionary and objective value walk into pipeline.py, which uses them at step 2 of the seven-step loop to decide which task each worker tackles next. Just as importantly, the same mathematical formula is reimplemented at line 43 of reward_functions.py as the per-step RL reward, so the allocator and the reward signal share the exact same definition of welfare — that alignment is why a policy trained to maximise cumulative reward ends up implicitly maximising the Nash product of surpluses.

core hcmarl/pipeline.py pipeline

Seven-step orchestration loop (Section 7.3): read state, allocate, translate load, compute neural drive, ECBF filter, integrate ODE, log.

If you want to know whether the four core modules agree on signs and units, this is the file that finds out. It opens by importing ECBFFilter, NSWFAllocator, ThreeCCr with get_muscle, and load_yaml with get_logger at lines 24–33, then defines a TaskProfile dataclass at line 43 (Def 7.1) that holds the per-muscle demand vector for each task. The body of the file is the seven-step loop from Section 7.3 of the math doc, and it runs in exactly that order: observe each worker's state x_i(t) = [MR, MA, MF], hand the state matrix to the NSWF allocator and let it pick a task per worker, translate each picked task into a per-muscle target load through the TaskProfile table, ask the baseline proportional controller (or, in training, the RL policy) for a nominal neural drive, project that drive through the ECBF QP to get C_safe, advance the 3CC-r ODE one tick under C_safe, and finally log everything that just happened. No optimisation runs in this file — that is the point. It is the trustworthy backbone that the RL stack later wraps.

Arrows in

Four imports converge on this file because the seven-step loop needs all four primitives in lockstep: ECBFFilter from ecbf_filter.py handles the safety projection at step 5, NSWFAllocator from nswf_allocator.py handles the assignment at step 2, ThreeCCr and get_muscle from three_cc_r.py drive the integration at step 6 and provide the muscle registry, and load_yaml with get_logger from utils.py handle config hydration and per-step diagnostics. Drop any one of those four and the loop cannot close.

Arrows out

What leaves this file is trajectory data — full (MR, MA, MF, C_safe, reward, cost) sequences — which walks straight into tests/. The test suite runs the pipeline end-to-end and asserts mass conservation, safety-set membership, and reward sign properties on the resulting trajectories, which is how a regression in any one of the four core modules surfaces as a pipeline test failure rather than a silent drift.

core hcmarl/mmicrl.py mmicrl

Multi-Modal Inverse Constrained RL: discovers latent worker types from demonstrations and learns per-type safety thresholds using normalizing flows.

This is the file that takes "workers are not interchangeable" seriously. The starting question is simple — given a pile of demonstrations from people with different bodies, can you discover the latent types they belong to and learn a different safety threshold for each? The answer is built out of a CFDE normalising flow at line 220, which is a stack of MADE/MAF blocks conditioned on a one-hot type vector z and trained to compute log p(x|z) for per-step features x = (state, action). On top of that flow runs an EM loop at line 332: the E-step at line 282 assigns each trajectory to its most likely type via argmax_z p(z|tau), using a tempered, mean-aggregated trajectory-level posterior at line 288 that prevents trajectory length from collapsing the assignments, and the M-step at line 369 then maximises −log p(x|z).mean() under the new assignments. Sitting above the flow is the MMICRL outer objective at line 1070 from Eq 10 — lambda2·I(tau;z) + (lambda1−lambda2)·H[pi(tau)] — which collapses to pure mutual information whenever the two lambdas are equal. Once types are settled, the ConstraintNetwork at line 468 takes over: it is a small MLP trained with binary cross-entropy at line 514 to discriminate expert states (label 1) from agent states (label 0), and the per-type, per-muscle theta_max is read off as the MF value where the network's cost output crosses 0.5. The DemonstrationCollector at line 549 is the bookkeeping piece that records (obs[agent].copy(), action) tuples per step per agent during rollouts (line 582), and fit() at line 1030 ties the whole sequence together: extract features, run CFDE EM, compute MI, learn the per-type constraint boundaries, and return the theta_per_type dictionary.

Arrows in

The only structural import comes through a lazy edge: real_data_calibration.py:962 reaches up and grabs DemonstrationCollector from this file at runtime. The lazy form is deliberate — it avoids a circular import while still letting the calibration module convert WSD4FEDSRM-anchored profiles into demonstrations that this file's collector can ingest, so the type discovery is grounded in real bodies rather than synthetic ones.

Arrows out

The headline output is a theta_per_type dictionary, and it walks straight into pettingzoo_wrapper.py, where the environment uses it at reset to set a personalised safety ceiling for each worker — no two workers in the same episode necessarily share a theta_max, which is the entire point of building MMICRL in the first place. Alongside that, the type assignments and likelihoods flow back into real_data_calibration.py for the downstream analysis of which WSD4FEDSRM subjects cluster into which latent type.

core hcmarl/real_data_calibration.py real_data

Path G: calibrates 3CC-r parameters against the WSD4FEDSRM dataset (34 participants, dynamic shoulder rotations) and produces correlated population samples.

This is the file that answers "are your muscle parameters real or convenient?" The answer starts with the WSD4FEDSRM dataset (Zenodo 8415066, Nature Scientific Data 2024) — thirty-four healthy participants performing dynamic shoulder internal/external rotation at 30–60% MVIC until maximal exhaustion (Borg RPE 20). For each subject, predict_endurance_time at line 44 simulates the 3CC-r ODE with the baseline proportional controller (Eq 35) at their recorded %MVC until the resting pool depletes below MR < 1e-4, and a grid search sweeps (F, R) to find the pair that minimises |ET_observed − ET_predicted|. That gives me per-subject, per-shoulder calibrated parameters anchored to measured data. For the five non-shoulder muscles — ankle, knee, elbow, trunk, grip — per-subject endurance data does not exist, so the POPULATION_FR dictionary at line 473 fills in with published isometric population means from Frey-Law et al. (2012) Table 2, and correlated sampling using CV_F at line 483 generates synthetic individual variation that preserves the inter-subject spread the literature reports. At line 962, the file lazy-imports DemonstrationCollector from mmicrl to convert these calibrated physiological profiles into (state, action) demonstrations, so that MMICRL's type-discovery pipeline runs on data grounded in real bodies rather than imagined ones.

Arrows in

MuscleParams, the ThreeCCr integrator, and the SHOULDER pre-calibrated instance all arrive from three_cc_r.py:37. Without them this file could not run the forward simulation that the grid-search calibration loops over — the calibration is literally fitting the parameters of the model defined in that module.

Arrows out

Two things leave. First, at line 962, a lazy import reaches up into mmicrl.py to grab DemonstrationCollector and feed the calibrated trajectories into the type-discovery pipeline, closing the loop between real-data physiology and latent-type inference. Second, the calibrated (F, R, r) tuples are baked into config/*.yaml, so that at training time the environment boots with real-data-grounded dynamics instead of the textbook population means.

env hcmarl/envs/warehouse_env.py warehouse

Single-worker discrete environment with action space Discrete(4), wrapping one 3CC-r body and one task.

This is the atomic unit — one human body, one task, one ThreeCCr instance. The action space is Discrete(4), and a call to reset() gives back an observation vector that concatenates the six muscles' (MR, MA, MF) triples with a task index; step(action) advances the ODE one tick and returns the new obs, reward, done flag, and info dict. There is no multi-agent logic here, no reward shaping, no coordination — just a clean interface around one body that does exactly one thing at a time.

Arrows in

Muscle state initialisation comes from three_cc_r — every call to reset() creates a fresh (MR=1, MA=0, MF=0) state, which is the fully-rested starting point defined by the ODE model.

Arrows out

The PettingZoo wrapper in pettingzoo_wrapper.py tiles N of these single-worker environments into one synchronous multi-agent environment. This box is the atomic brick that gets stacked — every worker in every experiment is one instance of this class.

env hcmarl/envs/pettingzoo_wrapper.py pettingzoo

PettingZoo parallel environment for N warehouse workers: bundles single-worker envs into one synchronous multi-agent step.

This is the contract that every method signs. It composes N single-worker environments into one PettingZoo parallel environment, and it enforces, by construction, that every method sees the same observation shape, the same reward function, and the same action space. For the default three-worker setup, each worker observes a 19-dimensional vector (obs_dim = 6 muscles × 3 compartments + 1 task index, line 60) and the centralised critic sees a 55-dimensional global state (global_obs_dim = 3 workers × 18 + 1, line 61). At construction time (lines 41–54), the wrapper accepts a per-worker, per-muscle theta_max dictionary — typically the output of MMICRL — and falls back to defaults (shoulder 0.70, ankle 0.80, etc.) if none is provided, so each worker runs with personalised safety ceilings from the start. The core physics step lives in _integrate at line 86, which advances the 3CC-r ODE with an inline ECBF analytical clipping pass that matches the coefficients in ecbf_filter.py exactly; this dual implementation keeps the environment self-contained while staying numerically aligned with the standalone filter. Every call to step(actions) at line 140 returns the standard PettingZoo dict-of-dicts — obs, rewards, terminations, truncations, infos — so the trainer never has to special-case anything.

Arrows in

Four things converge here: nswf_reward and safety_cost from reward_functions.py:7, which the wrapper calls inside every step() to produce the reward and cost signals that the agents learn from; get_muscle from three_cc_r.py:32, called at construction to initialise the per-muscle (F, R, r) dictionaries for the ODE; TaskProfileManager from task_profiles.py at line 17, which translates a discrete action (a task id) into the per-muscle demand vector the ODE consumes; and the per-worker theta_max dictionary from mmicrl.py, which personalises the safety ceiling so the ECBF inside _integrate clips to the right boundary for each worker.

Arrows out

The constructed environment instance is the single object that scripts/train.py and scripts/evaluate.py instantiate. Every training method and every evaluation run — all seven methods, all ten baselines, all ablations and scaling sweeps — passes through this one wrapper. If the observation shape, reward formula, or action space ever drifted between methods, the entire comparative evaluation would be meaningless.

env hcmarl/envs/reward_functions.py reward_fn

Pure functions for NSWF-shaped reward and safety cost — no state, no class, testable in isolation.

Three pure functions, no state, no class — deliberately kept stateless so they can be unit-tested in complete isolation. disagreement_utility at line 33 computes D_i(MF) = kappa·MF²/(1−MF) from Eq 32 — the same formula the allocator uses, repeated here so the per-step reward signal stays mathematically aligned with the allocation objective. nswf_reward at line 43 builds on that by returning log(productivity − D_i) per step, which means maximising cumulative RL return over an episode is algebraically equivalent to maximising the Nash product of surpluses from Eq 33. safety_cost at line 81 is the constraint side: it returns a non-negative cost whenever any muscle's MF exceeds its theta_max, and this is the number that the Lagrangian multiplier in MAPPO-Lag penalises and that the logger counts as a violation.

Arrows in

None. These functions import nothing from the codebase — they take floats in and return floats out, which is why they can be tested without constructing any environment or agent.

Arrows out

Both nswf_reward and safety_cost are imported at line 7 of pettingzoo_wrapper.py, and the wrapper calls them inside every step() to produce the reward and cost signals that flow into every agent's rollout buffer.

agent hcmarl/agents/networks.py networks

Three small MLPs with orthogonal init: ActorNetwork, CriticNetwork, CostCriticNetwork.

Three small MLPs, all orthogonally initialised, all deliberately undersized because the bottleneck in this problem is data, not capacity. ActorNetwork at line 16 takes the worker's 19-dimensional local observation through two 64-unit Tanh layers and outputs logits over a Categorical distribution; get_action at line 35 returns the sampled action, its log-probability, and the entropy of the distribution in one call, so the PPO update has everything it needs without a second forward pass. CriticNetwork at line 43 takes the 55-dimensional global state through two 128-unit Tanh layers and produces a scalar V(s), while CostCriticNetwork at line 59 mirrors that architecture exactly to produce V_c(s) for discounted cost. The output layer of the actor is initialised with gain 0.01 (small initial logits, nearly uniform policy at the start), while hidden layers use gain sqrt(2) — the standard recipe from Andrychowicz et al. (2020) for PPO-style training.

Arrows in

None. These are leaf definitions — they import only PyTorch and they depend on nothing in the codebase, which is why any agent class can import them without pulling in the rest of the framework.

Arrows out

All three networks flow into mappo.py:14, mappo_lag.py:11, and ippo.py:11. The split is clean: MAPPO instantiates one actor and one value critic, MAPPO-Lag adds the cost critic on top, and IPPO creates per-agent copies of the actor and value critic, each using local observations instead of the global state.

agent hcmarl/agents/mappo.py mappo

Centralised-critic, decentralised-actor PPO (CTDE paradigm) with GAE and a shared rollout buffer.

Centralised training, decentralised execution — that is the idea, and this file is where it materialises. The RolloutBuffer at line 17 stores per-agent observations, actions, log-probabilities, rewards, and done flags alongside a shared centralised value estimate per timestep, and at update time it flattens everything into (T×N, …) tensors at line 153 so the PPO gradients mix across agents and timesteps simultaneously. compute_returns at line 114 computes generalised advantage estimates per agent (gamma=0.99, lambda=0.95) and interleaves them into the same flat array. The actor loss at line 254 is the standard PPO clipped surrogate — −min(r·A, clip(r, 1±eps)·A).mean() − beta·H.mean() — and the entropy bonus prevents premature collapse to a deterministic policy. The critic loss at line 261 is plain MSE(V(s), GAE_returns). Both networks are updated under gradient clipping at max_grad_norm=0.5, which keeps the training stable without needing aggressive learning-rate scheduling.

Arrows in

ActorNetwork and CriticNetwork arrive from networks.py:14. MAPPO instantiates one shared actor and one centralised critic from these classes — the reason it needs the global-obs critic is that decentralised actors trained against only local information cannot coordinate on a cooperative task.

Arrows out

Three consumers depend on this file. First, hcmarl_agent.py:8 wraps the MAPPO class as its policy backbone — the integrated HC-MARL agent is MAPPO plus ECBF plus NSWF, not a replacement for MAPPO. Second, ippo.py:12 imports RolloutBuffer directly to avoid duplicating the GAE computation. Third, train.py:26 imports MAPPO as one of the seven methods its factory can return.

agent hcmarl/agents/ippo.py ippo

Independent PPO: each worker trains its own actor-critic with no centralised information. Deliberately weak baseline.

IPPO is the deliberately weak baseline — every worker trains in a silo, sees only its own observation, and has no idea what the other workers are doing. Lines 31–32 instantiate n_agents separate copies of both ActorNetwork and CriticNetwork, and each critic uses obs_dim (local) rather than global_obs_dim, so coordination can only emerge through the shared environment dynamics, never through the value function. The actor loss at line 87 is the same PPO clipped surrogate as MAPPO, just with a per-agent local value baseline, and the critic loss at line 95 is per-agent MSE(V_i(o_i), returns_i). The file reuses RolloutBuffer from mappo.py:12 — one buffer per agent — to avoid duplicating the GAE logic.

Arrows in

ActorNetwork and CriticNetwork from networks.py:11, each cloned per agent, and RolloutBuffer from mappo.py:12, reused to keep the GAE computation in one place.

Arrows out

IPPO flows to train.py as one of the seven factory options. The reason it exists in the comparison is simple: if a learned method cannot outperform independent agents who share nothing, the coordination mechanism is not earning its keep.

agent hcmarl/agents/mappo_lag.py mappo_lag

Lagrangian MAPPO: same actor + two critics (value and cost) + a learnable multiplier lambda that pushes cost below a budget.

This is MAPPO with a safety budget. Everything from the base MAPPO class is here — shared actor, centralised critic, PPO clipped surrogate — but the actor loss at line 237 picks up an additional term: lambda · max(cost_surr1, cost_surr2).mean(). The max (rather than min) means the cost penalty is conservatively clipped — under ratio truncation the policy errs on the side of safety, not reward. That lambda is itself a learnable quantity: log_lambda is an nn.Parameter at line 169 with its own Adam optimiser at line 176, updated at line 270 via the dual gradient −exp(log_lambda)·(mean_cost − cost_limit). When the average cost overshoots the budget (default cost_limit=0.1), lambda rises and pushes the policy harder toward safety; when the constraint is satisfied, lambda decays and lets the policy chase reward again. Alongside the actor, the reward critic at line 246 (MSE(V(s), reward_returns)) and the cost critic at line 254 (MSE(V_c(s), cost_returns)) train in parallel — the cost critic is what gives the actor a gradient signal about future cost, not just the cost it has already incurred.

Arrows in

All three network classes — ActorNetwork, CriticNetwork, CostCriticNetwork — arrive from networks.py:11. This is the only agent in the codebase that needs all three, because it is the only one that tracks both reward returns and cost returns separately.

Arrows out

MAPPOLagrangian flows in two directions. First, safepo_wrapper.py:12 imports it directly so that SafePO's comparison runs use the exact same Lagrangian numerics as the in-house implementation — keeping the comparison on equal numerical footing. Second, train.py imports it as one of the seven factory options.

agent hcmarl/agents/hcmarl_agent.py hcmarl_agent

The integrated method: MAPPO backbone + ECBF safety projection + NSWF reward + per-worker theta_max from MMICRL.

This is the only file where all four ideas meet in one constructor: MAPPO as the policy backbone, ECBF parameters for safety, NSWF as the reward structure, and the MMICRL-derived per-worker theta_max dictionary that personalises the safety ceiling. The constructor at line 12 accepts (obs_dim, global_obs_dim, n_actions, n_agents, theta_max=None, ecbf_params=None, use_nswf=True, device='cpu') and wraps a MAPPO instance internally at line 14 — it composes MAPPO rather than reimplementing it, so the actor, critic, buffer, and update logic are all inherited without duplication. The key architectural decision is that the agent proposes actions through get_actions at line 23, which delegates straight to self.mappo.get_actions(), but the actual safety filtering happens inside the environment's _integrate method, not here. The agent never clips itself — the environment does that, so the RL gradient remains clean and the safety guarantee lives in the physics layer rather than the policy layer.

Arrows in

MAPPO arrives from mappo.py:8, and it arrives as a composed member, not a parent class — every piece of the policy, buffer, and update mechanism comes from that single import.

Arrows out

HCMARLAgent flows to train.py as the primary method in the factory. It is the method the framework is built to test — every other method in the comparison exists to measure what happens when you take away one or more of its four components.

agent hcmarl/baselines/_legacy.py legacy

Ten classical baselines (random, round-robin, greedy, greedy-safe, PPO-unconstrained, PPO-Lagrangian, MAPPO-NoFilter, MAPPO-Lagrangian, MACPO, fixed-schedule) in one file.

Ten classical baselines in one file, defined at lines 28, 44, 64, 79, 110, 145, 196, 224, 236, and 275: Random, Round-Robin, Greedy, Greedy-Safe, PPO-Unconstrained, PPO-Lagrangian, MAPPO-NoFilter, MAPPO-Lagrangian, MACPO, and Fixed-Schedule. Phase 1 — Why it exists

Every other module depends on the muscle ODE being right. If the integrator drifts mass or the parameters are fabricated, every reward, every safety guarantee, every figure in the paper is downstream of a lie. I keep this phase small on purpose so I can read every line of it.

Phase 2 — Why it matters

The ECBF turns "hopefully safe" into "safe by construction" — Nagumo invariance (Theorem 5.7) gives me a theorem, not a hope. The NSWF allocator turns "fair" into "log-product of surpluses" — also a theorem (Nash 1950, Kaneko & Nakamura 1979). Phase 2 is where the math stops being decorative and starts being load-bearing.

Phase 3 — Why it is the trust anchor

If pipeline.py runs end-to-end without the trainer touching it, I know the physics, the safety filter, and the allocator agree on signs and units. Every later bug becomes a learning bug, not a modelling bug. That separation is what makes debugging tractable.

Phase 4 — Why it is the legitimacy phase

A reviewer will ask whether the muscle parameters are real or fitted to look convenient. Path G against WSD4FEDSRM — 34 real participants, dynamic shoulder rotation, measured endurance times — is the answer. MMICRL exists because workers are not interchangeable: handing every worker the same theta_max silently averages over very different bodies. Latent types let the safety filter be personal.

Phase 5 — Why it is the contract

The PettingZoo wrapper is the only file every method has to agree with. If obs_dim, global_obs_dim, action space, or reward shape drifted between methods, the entire comparative evaluation would be meaningless. This phase is short, but its API is non-negotiable.

Phase 6 — Why it is honest

I keep the networks small because the bottleneck is data, not model capacity. The Lagrangian agent earns its additional complexity only because the cost critic and the multiplier update solve a problem — staying under a safety budget — that the plain value critic cannot.

Phase 7 — Why it is the comparison

Without the OmniSafe and SafePO wrappers I would be comparing my method only against itself with components removed. With them, I am comparing against the field's established constrained MARL methods, on the same environment, the same seeds, the same logger.

Phase 8 — Why it is one file

One train.py with one factory means the seven methods share the same buffer logic, the same logging, the same seeding. Every alternative I have seen — per-method scripts, inheritance hierarchies — drifts within a month.

Phase 9 — Why it reuses the training graph

Importing create_agent from train.py into evaluate.py is the simplest way to guarantee the evaluation graph never lags behind the training graph. One source of truth for what each method actually is.

Phase 10 — Why it is the safety net

Configs in YAML, notebooks read-only, 223 tests pinned. These are not interesting. They are what prevents discovering, the night before submission, that a number drifted three weeks ago and no one noticed.

Objective functions (exhaustive audit)

Every optimisation problem in the codebase, listed one by one. Re-grepped across all files in hcmarl/ and scripts/.

1. PPO clipped surrogate — MAPPO actor

hcmarl/agents/mappo.py:254
L_actor = -min(r(theta)*A, clip(r(theta), 1-eps, 1+eps)*A).mean() - beta * H.mean()

Optimises: The policy parameters theta to maximise expected advantage while staying close to the old policy (trust region via clipping).

Why it exists: Standard PPO objective. Entropy bonus beta*H prevents premature convergence.

2. Value critic MSE — MAPPO

hcmarl/agents/mappo.py:261
L_critic = MSE(V(s), R_GAE)

Optimises: The centralised critic's parameters to predict discounted returns computed via GAE (gamma=0.99, lambda=0.95).

Why it exists: Accurate value estimates reduce variance in the policy gradient.

3. PPO clipped surrogate — IPPO actor (per-agent)

hcmarl/agents/ippo.py:87
L_actor_i = -min(r_i*A_i, clip(r_i)*A_i).mean() - beta * H_i.mean()

Optimises: Each worker's independent policy parameters. Same loss shape as MAPPO, but with a per-agent local value baseline.

Why it exists: IPPO serves as a deliberately weak baseline — no centralised information, no coordination.

4. Value critic MSE — IPPO (per-agent, local)

hcmarl/agents/ippo.py:95
L_critic_i = MSE(V_i(o_i), R_i)

Optimises: Each worker's local critic using only local observations. No global state access.

Why it exists: Independent baseline — measures the value of centralisation by its absence.

5. Lagrangian actor loss — MAPPO-Lag

hcmarl/agents/mappo_lag.py:237
L_actor = -min(surr1, surr2).mean() + lambda * max(cost_surr1, cost_surr2).mean() - beta * H.mean()

Optimises: The policy to maximise reward while penalising constraint violations. The cost term uses max (conservative clipping) to stay safe under ratio truncation.

Why it exists: Constrained CMDP formulation. Lambda scales how aggressively the cost penalty pushes the policy away from unsafe states.

6. Reward critic MSE — MAPPO-Lag

hcmarl/agents/mappo_lag.py:246
L_critic = MSE(V(s), R_GAE)

Optimises: Centralised value critic, same as MAPPO.

7. Cost critic MSE — MAPPO-Lag

hcmarl/agents/mappo_lag.py:254
L_cost_critic = MSE(V_c(s), C_GAE)

Optimises: The cost critic's parameters to predict discounted cumulative cost.

Why it exists: The Lagrangian actor needs an estimate of future cost to know whether it is currently violating the budget.

8. Lagrangian dual variable update

hcmarl/agents/mappo_lag.py:270
L_lambda = -exp(log_lambda) * (mean_cost - cost_limit)

Optimises: The Lagrangian multiplier lambda. Gradient ascent: lambda increases when mean_cost > cost_limit, decreases when the constraint is satisfied.

Why it exists: Dual gradient ascent on the constrained optimisation. Default cost_limit = 0.1.

9. CFDE flow negative log-likelihood — MMICRL M-step

hcmarl/mmicrl.py:369 (also line 862 for the full-pipeline variant)
L_flow = -log p(x|z).mean()

Optimises: The MADE/MAF flow parameters to maximise the conditional density of per-step features x given type assignment z.

Why it exists: The M-step of EM. After types are assigned (E-step), the flow is trained to model each type's state-action distribution.

10. EM type assignment — MMICRL E-step

hcmarl/mmicrl.py:286 (per-step), line 328 (trajectory-level)
z* = argmax_z p(z|tau) where p(z|tau) = softmax(log p(z) + mean_{(s,a)} log p(s,a|z))

Optimises: The assignment of demonstrations to latent types. Hard MAP assignment.

Why it exists: The E-step of EM. Trajectory-level aggregation (mean, not sum) prevents length-dependent mode collapse.

11. MMICRL outer objective (Eq 10)

hcmarl/mmicrl.py:1070
J = lambda2 * I(tau; z) + (lambda1 - lambda2) * H[pi(tau)]

Optimises: The overall MMICRL objective — mutual information between trajectories and types, plus a marginal entropy term. When lambda1 = lambda2, this reduces to pure MI.

Why it exists: This is the top-level objective from Qiao et al. that the EM procedure maximises. It measures how informative the type assignments are.

12. ConstraintNetwork BCE

hcmarl/mmicrl.py:514
L_constraint = BCE(c_theta(s), label) where label=1 for expert, label=0 for agent

Optimises: A small MLP to distinguish expert states (safe) from agent states (potentially unsafe). The decision boundary at c_theta = 0.5, swept across MF values, yields theta_max per muscle.

Why it exists: Learns the constraint boundary from demonstrations rather than hand-tuning theta_max.

13. ECBF quadratic program

hcmarl/ecbf_filter.py:270
C* = argmin_{C >= 0} ||C - C_nom||^2 s.t. ECBF(Eq 18) and CBF(Eq 23)

Optimises: The neural drive C at each timestep. Finds the closest safe control to the policy's nominal proposal.

Why it exists: Hard safety guarantee. The QP enforces both barriers (fatigue ceiling and resting floor) simultaneously, at every step, regardless of what the policy proposes.

14. NSWF allocation objective (Eq 33)

hcmarl/nswf_allocator.py:255–266
a* = argmax_{a} sum_i log(U(i, a_i) - D_i(MF_i))

Optimises: The task assignment to maximise the Nash Social Welfare — the product of worker surpluses in log space.

Why it exists: Guarantees fairness: no worker can be assigned a task whose surplus is below their fatigue-driven disagreement threshold. The log transform makes the product compatible with RL's additive reward structure.

15. Path G calibration — grid search

hcmarl/real_data_calibration.py (predict_endurance_time, line 44)
(F*, R*) = argmin_{F, R} |ET_observed - ET_predicted(F, R, r, target_load)|

Optimises: Per-subject (F, R) parameters to match observed endurance times from the WSD4FEDSRM dataset (34 participants, dynamic shoulder rotation).

Why it exists: Anchors the simulator to real human fatigue data. Without this, the 3CC-r parameters would be untested population means, not calibrated individual fits.

16. k-means warm start for type centroids

hcmarl/mmicrl.py:413
z_i = argmin_k ||x_i - mu_k||^2

Optimises: Initial cluster assignments before EM begins. Standard k-means on the feature space.

Why it exists: EM is sensitive to initialisation. k-means provides a warm start that prevents degenerate initial assignments (e.g. all trajectories assigned to one type).

Performance measures (exhaustive)

The nine metrics in logger.py:17–21, computed by compute_episode_metrics (line 65). Plus derived metrics.

1. violation_rate lower is better

logger.py:73
violation_rate = total_violations / (n_steps * n_workers * n_muscles)

Fraction of (step, worker, muscle) slots where MF > theta_max. A method with violation_rate = 0 never breaches any safety threshold.

2. cumulative_cost lower is better

logger.py:74
cumulative_cost = total_violations (count)

Raw count of safety violations across the episode. This is the quantity the Lagrangian multiplier lambda penalises.

3. safety_rate higher is better

logger.py:75
safety_rate = safe_steps / n_steps

Fraction of timesteps where no worker violates any threshold. Complement of per-step violation frequency.

4. tasks_completed higher is better

logger.py:76
tasks_completed = count of productive tasks completed in the episode

Measures throughput. A method that rests all workers achieves safety_rate = 1 but tasks_completed = 0.

5. cumulative_reward higher is better

logger.py:77
cumulative_reward = sum of nswf_reward across all steps and workers

The RL return. Shaped by the NSWF log-surplus, so maximising this is equivalent to maximising the Nash product of surpluses (Eq 33).

6. jain_fairness higher is better

logger.py:82
jain_fairness = (sum(tpw))^2 / (n * sum(tpw^2))

Jain's fairness index over per-worker task counts. Value in [1/n, 1]. jain_fairness = 1 means every worker completed the same number of tasks.

7. peak_fatigue lower is better

logger.py:83
peak_fatigue = max MF across all workers and muscles during the episode

Worst-case fatigue level observed. If ECBF is working, this should stay below theta_max.

8. forced_rest_rate lower is better

logger.py:84
forced_rest_rate = forced_rests / (n_steps * n_workers)

Fraction of (step, worker) slots where the ECBF forced the worker to rest. High values mean the policy is proposing unsafe actions that the filter has to override.

9. constraint_recovery_time lower is better

logger.py:86
constraint_recovery_time = mean(recovery_times) in steps

Average number of steps to recover from a constraint violation (MF dropping back below theta_max). Shorter is better.

Novelty

  • The integration of a relative-degree-2 ECBF filter, a Nash Social Welfare reward shaping, MMICRL latent worker types, and Lagrangian MAPPO inside a single multi-agent learning loop. None of the four ideas is new in isolation. The wiring is.
  • The dual barrier formulation (h = theta_max - MF and h2 = MR >= 0) on the 3CC-r ODE, with rest-phase safety proved via Nagumo invariance (Theorem 5.7). This specific construction — applying ECBF to a physiological fatigue model — has not been done before.
  • The per-worker, per-muscle theta_max dictionary produced by MMICRL and consumed by the safety filter. This personalises safety thresholds by latent type discovered from demonstrations, rather than by demographic proxy.
  • The Path G calibration loop against WSD4FEDSRM, which ties the simulator's muscle parameters to real measured endurance times from 34 participants rather than relying on untested population means.
